Celery Exporter provides two pre-built Grafana dashboards hosted on Grafana.com. This page explains what each dashboard shows, how to import them into your Grafana instance, which metrics they depend on, and how to use the available template variables to filter data.

## Available dashboards

<CardGroup cols={2}>
  <Card title="Celery Tasks by Task" icon="chart-bar" href="https://grafana.com/grafana/dashboards/17508">
    Dashboard ID **17508**. Breaks down task throughput, failure rates, and runtime histograms per individual task name. Use this to identify which tasks are slow or failing.
  </Card>
  <Card title="Celery Tasks Overview" icon="chart-line" href="https://grafana.com/grafana/dashboards/17509">
    Dashboard ID **17509**. Shows aggregate queue depths, worker counts, and overall task throughput. Use this as your top-level health view.
  </Card>
</CardGroup>

## Importing dashboards into Grafana

<Steps>

<Step title="Open the import dialog">

In your Grafana instance, navigate to **Dashboards → New → Import**.

</Step>

<Step title="Enter the dashboard ID">

In the **Import via grafana.com** field, enter the dashboard ID and click **Load**:

<CodeGroup>

```text Celery Tasks by Task
17508
```

```text Celery Tasks Overview
17509
```

</CodeGroup>

Grafana fetches the dashboard JSON from Grafana.com automatically.

</Step>

<Step title="Select your Prometheus data source">

On the import confirmation screen, select the Prometheus data source that is scraping your Celery Exporter instance. Then click **Import**.

</Step>

</Steps>

<Note>
Both dashboards are also available as JSON files generated by the Celery mixin. See the [Celery mixin dashboards page](/mixin/dashboards) for instructions on generating customized versions locally.
</Note>

## Dashboard details

### Celery Tasks by Task (ID 17508)

This dashboard is oriented around individual task names. It is most useful when you want to debug a specific task type or compare performance across tasks.

**Panels included:**

- **Task throughput** — rate of `celery_task_succeeded_total` and `celery_task_failed_total` per task name over time, showing completions per second.
- **Failure rate** — percentage of tasks that failed, calculated as `failed / (failed + succeeded)`, grouped by task name.
- **Runtime histogram** — `celery_task_runtime_bucket` visualized as a heatmap, showing the distribution of execution times per task. Useful for spotting latency regressions.
- **Retry rate** — rate of `celery_task_retried_total` per task name.

### Celery Tasks Overview (ID 17509)

This dashboard is a high-level operational view. It is most useful as a first stop when investigating an incident or reviewing overall system health.

**Panels included:**

- **Queue depth** — current `celery_queue_length` per queue, showing how many tasks are waiting to be processed.
- **Worker count** — `celery_worker_up` and `celery_active_worker_count` to show how many workers are alive and subscribed to each queue.
- **Active process count** — `celery_active_process_count` per queue, showing total worker concurrency.
- **Overall throughput** — aggregate task completion and failure rates across all task names.

## Template variables

Both dashboards expose the following template variables in the top toolbar. Use them to filter the panels to the data you care about.

| Variable | Description |
|----------|-------------|
| `job` | Filters by the Prometheus `job` label. Set to the job name you used in your `prometheus.yml` scrape config (e.g., `celery`). |
| `queue_name` | Filters all panels to a specific broker queue. Useful when you have multiple queues (e.g., `default`, `priority`, `emails`). |
| `task` | Filters task-level panels to a specific task name. Available in the Tasks by Task dashboard. |

<Tip>
Set the `job` variable to match the `job_name` in your `prometheus.yml` scrape config. If you followed the quickstart, this is `celery`. Mismatched job names will result in empty panels.
</Tip>

## Required metrics

The dashboards rely on the following metrics being present in Prometheus. If any are missing, the corresponding panels will show no data.

| Dashboard | Required metrics |
|-----------|-----------------|
| Tasks by Task | `celery_task_succeeded_total`, `celery_task_failed_total`, `celery_task_retried_total`, `celery_task_runtime_bucket` |
| Tasks Overview | `celery_queue_length`, `celery_worker_up`, `celery_active_worker_count`, `celery_active_process_count`, `celery_task_succeeded_total`, `celery_task_failed_total` |

<Warning>
`celery_task_runtime_bucket` only appears after at least one task has succeeded. The runtime heatmap panel will be empty until the first successful task is recorded.
</Warning>

## Generating custom dashboards via the Celery mixin

The Grafana.com dashboards are generated from the Celery mixin using Jsonnet and Grafonnet. If you need to customize panel queries, change thresholds, or adapt the dashboards to your environment, generate them locally from the mixin source.

See the [Celery mixin dashboards page](/mixin/dashboards) for full instructions on building customized dashboard JSON files.
